# Каталог моделей и запасные модели

Каталог — это описание того, чем вы работаете. В нём две главные части: **провайдеры** (куда стучаться и чем авторизоваться) и **модели** (что именно звать и с какими настройками).

Ключевая идея: единица выбора — модель, а не провайдер. Провайдер знает только адрес и токен; модель знает свой слаг, свои параметры и свою политику при сбоях. Поэтому одна и та же модель у двух провайдеров — это две записи каталога с разными ключами, и ничего не путается.

## Растём от минимального конфига

Начинаем с того, без чего нельзя:

```php
$registry = Registry::fromArray([
    'providers' => [
        'openai' => ['class' => OpenAiProvider::class, 'token' => 'sk-...'],
    ],
    'models' => [
        'mini' => ['provider' => 'openai', 'name' => 'gpt-4o-mini'],
    ],
    'defaultModel' => 'mini',
]);
```

`mini` — это **ключ каталога**: он попадает в ваш интерфейс и в вашу базу. `gpt-4o-mini` — **слаг**, под которым модель знает API провайдера. Разделение нужно, потому что слаги у разных шлюзов различаются, а ключ остаётся вашим.

Добавим вторую модель, подписи для интерфейса и параметры генерации:

```php
'models' => [
    'mini' => [
        'provider'    => 'openai',
        'name'        => 'gpt-4o-mini',
        'fullName'    => 'GPT-4o mini',
        'description' => 'быстрая и дешёвая, для простых задач',
        'params'      => ['temperature' => 0.2, 'maxTokens' => 2000],
    ],
    'sonnet' => [
        'provider'    => 'openrouter',
        'name'        => 'anthropic/claude-3.5-sonnet',
        'fullName'    => 'Claude 3.5 Sonnet',
        'description' => 'для сложных задач и длинной истории',
    ],
],
```

Теперь `$registry->labels()` отдаст `['mini' => 'GPT-4o mini', 'sonnet' => 'Claude 3.5 Sonnet']` — готовый список для выпадающего меню.

И, наконец, скажем, что делать при сбоях:

```php
'fallback'      => ['mini', 'sonnet'],     // порядок, в котором пробуем другие модели
'maxSwitches'   => 2,                      // не больше двух переключений за вызов
'defaultPolicy' => ['retries' => 2, 'delay' => 5, 'backoff' => 2],
```

Это весь каталог. Дальше — что ещё можно в нём указать.

## Провайдер

```php
'providers' => [
    'openrouter' => [
        'class'   => OpenRouterProvider::class,
        'token'   => 'sk-or-...',
        'timeout' => 180,
        'headers' => ['HTTP-Referer' => 'https://example.com', 'X-Title' => 'My App'],
    ],
],
```

Что можно задать:

- **`class`** (обязательно) — реализация `ProviderInterface`. В комплекте: `OpenAiProvider`, `OpenRouterProvider`, `RequestyProvider` — они говорят по одному и тому же OpenAI-совместимому API и отличаются адресом и именем. Свой провайдер — [12-custom-provider.md](12-custom-provider.md).
- **`token`** — ключ авторизации. Пустой допустим: локальные шлюзы его не требуют.
- **`baseUrl`** — адрес API без пути к эндпоинту. По умолчанию берётся из класса провайдера; задавайте, если работаете через прокси или self-hosted шлюз.
- **`timeout`** — таймаут запроса в секундах, по умолчанию `120`. Модель может перекрыть его своим.
- **`headers`** — заголовки для всех запросов этого провайдера.
- **`extraParams`** — дополнительные поля payload для всех его запросов.
- **`capture`** — карта извлечения полей ответа, о ней ниже.
- **`debug`** — при `true` payload и сырой ответ пишутся в лог на уровне `debug`.
- **`keepRaw`** — при `false` сырой ответ не сохраняется в `Response`. По умолчанию сохраняется.
- **`httpClient`** — свой HTTP-клиент, объект или фабрика, см. [11-custom-http-client.md](11-custom-http-client.md).
- **`meta`** — любые данные приложения; библиотека их не трогает.

## Модель

```php
'glm' => [
    'provider'    => 'requesty',
    'name'        => 'zai/GLM-4.6',
    'fullName'    => 'GLM-4.6',
    'description' => 'быстрая, аккуратная с инструментами',
    'params'      => ['temperature' => 0.2],
],
```

Что можно задать:

- **`provider`** (обязательно) — ключ провайдера из этого же каталога.
- **`name`** (обязательно) — слаг модели для API.
- **`fullName`** — название для интерфейса. По умолчанию равно ключу.
- **`description`** — чем модель хороша; удобно показывать в списке выбора.
- **`params`** — `temperature`, `topP`, `maxTokens`, `seed`.
- **`unsupported`** — параметры, которые модель не принимает. Вырезаются из запроса всегда, кто бы их ни задал.
- **`extraParams`** — поля payload, специфичные для этой модели.
- **`headers`** — заголовки, специфичные для этой модели.
- **`capture`** — карта извлечения полей ответа поверх карты провайдера.
- **`policy`** — своя политика повторов, см. ниже.
- **`timeout`** — свой таймаут в секундах; полезен медленным рассуждающим моделям.
- **`pricing`** — `['in' => 1.25, 'out' => 10.0]`, цена за миллион токенов, для оценки стоимости.
- **`tags`** — метки для группировки в приложении, например `['fast', 'tools']`.
- **`meta`** — любые данные приложения; библиотека их не трогает.

### Один слаг, два режима

Две записи с одинаковым `name`, но разными `extraParams` — это два пункта в списке выбора. Ключ сохраняется у вас в базе, поэтому потом видно, в каком режиме отвечала модель.

```php
'glm'       => ['provider' => 'requesty', 'name' => 'zai/GLM-4.6', 'fullName' => 'GLM-4.6'],
'glm-think' => [
    'provider'    => 'requesty',
    'name'        => 'zai/GLM-4.6',
    'fullName'    => 'GLM-4.6 (глубокое мышление)',
    'extraParams' => ['thinking' => ['type' => 'enabled']],
    'params'      => ['maxTokens' => 16000],
],
```

### Модель, которая не принимает семплирование

Рассуждающие модели обычно отвергают `temperature` и `top_p`, зато принимают собственные поля и отвечают дольше:

```php
'gpt-5' => [
    'provider'    => 'openai',
    'name'        => 'gpt-5',
    'unsupported' => ['temperature', 'topP'],
    'extraParams' => ['reasoning_effort' => 'high'],
    'timeout'     => 600,
],
```

Перечисляйте в `unsupported` именно то, что модель не принимает. Соседний случай — когда параметры поддерживаются, но не вместе: часть провайдеров отвечает «temperature and top_p cannot both be specified». Специального механизма для этого не нужно: незаданные параметры в запрос не попадают, а `topP` по умолчанию нигде не задан, поэтому конфликт возникает, только если вы сами укажете оба. Тогда достаточно убрать лишний — из `params` или через `unsupported` у этой модели.

## Откуда берётся итоговый запрос

Настройки задаются на трёх уровнях, и это главное, что нужно понять про каталог.

Параметры генерации (`temperature`, `topP`, `maxTokens`, `seed`) сливаются **по явности**: каталог (`defaultParams`) → модель (`params`) → конкретный вызов. Чем ближе к вызову, тем сильнее. Поверх результата применяется `unsupported`: он не пожелание, а ограничение модели, поэтому побеждает всех.

Произвольные поля payload и заголовки сливаются в том же порядке: провайдер → модель → вызов. Правила слияния:

- ассоциативные массивы сливаются вглубь — `reasoning.effort` не затирает `reasoning.enabled`;
- списки заменяются целиком — `provider.order` это выбор, а не накопление;
- значение `null` удаляет ключ: так модель отменяет то, что задал её провайдер.

```php
// провайдер: 'extraParams' => ['provider' => ['order' => ['DeepInfra']]]
// модель:    'extraParams' => ['thinking' => ['type' => 'enabled']]
// вызов:     $config->extraParams = ['session_id' => 'run_17'];
// в запрос уйдут все три поля
```

## Политика повторов

Уровень повторов в библиотеке ровно один — этот. Транспорт своих циклов не крутит, поэтому время ожидания при сбое предсказуемо.

```php
'defaultPolicy' => [
    'retries'     => 2,
    'delay'       => 5,
    'backoff'     => 2,
    'then'        => 'fallback',
    'perCategory' => [ErrorCategory::RATE_LIMIT => ['retries' => 3, 'delay' => 15]],
],
```

- **`retries`** — сколько дополнительных попыток делать той же моделью. По умолчанию `2`, то есть всего три вызова.
- **`delay`** — базовая пауза перед повтором в секундах, по умолчанию `5`.
- **`backoff`** — во сколько раз растёт пауза с каждой попыткой, по умолчанию `2`: получается 5 с, потом 10 с.
- **`maxDelay`** — потолок одной паузы, по умолчанию `60` секунд.
- **`then`** — что делать, когда повторы кончились: `'fallback'` (передать работу следующей модели) или `'stop'` (вернуть ошибку). По умолчанию `'fallback'`.
- **`perCategory`** — переопределения для отдельных категорий ошибок. Типичный случай — ограничение частоты: ждать дольше и настойчивее.
- **`retryOn`** — явный список категорий, которые повторяем. Если пусто, решает сама категория.
- **`stopOn`** — категории, при которых не переключаемся на запасную модель, даже если `then` разрешает.
- **`maxWaitSeconds`** — потолок времени на эту модель: её запросы плюс паузы между повторами. По умолчанию не ограничено.

### Своя политика у модели и у провайдера

Ту же секцию можно положить провайдеру и отдельной модели — ключ везде называется `policy`:

```php
'providers' => [
    'slow-gateway' => [
        'class'  => OpenRouterProvider::class,
        'token'  => 'sk-or-...',
        'policy' => ['retries' => 1, 'delay' => 10],     // для всех моделей этого шлюза
    ],
],

'models' => [
    'gpt-5' => [
        'provider' => 'openai',
        'name'     => 'gpt-5',
        'policy'   => ['retries' => 1, 'then' => 'stop'],  // чтобы дорогую модель не подменяли молча
    ],
],
```

**Политика не собирается по кусочкам: действует та, что задана ближе всего, и действует целиком.** Порядок поиска: политика прогона (`Config::$policy` или `Orchestra::withPolicy()`) → `policy` модели → `policy` её провайдера → `defaultPolicy` каталога. Первая найденная и применяется; остальные не подмешиваются.

Отсюда два следствия.

Поля, которые в найденной политике не заданы, берут значения по умолчанию класса `ErrorPolicy` (`retries = 2`, `delay = 5`, `backoff = 2`, `maxDelay = 60`, `then = fallback`), а не значения из каталога. В примере выше у `gpt-5` пауза будет 5 секунд, даже если в `defaultPolicy` стоит другая.

Уточнения по категориям тоже принадлежат своей политике целиком. Если модель задала `policy`, каталожные `perCategory` на неё не действуют — нужные правила перечисляются в её же секции:

```php
'policy' => [
    'retries'     => 3,
    'perCategory' => [ErrorCategory::TIMEOUT => ['retries' => 0]],
],
```

Такой размен сделан осознанно: настройка повторяется, зато по конфигу сразу видно, как поведёт себя конкретная модель, и не приходится держать в голове, что перекрывает что.

Обратите внимание, у кого что спрашивают: `retries`, `delay` и `backoff` берутся у модели, которая выполняется сейчас, а решение «переключаться или остановиться» (`then`, `stopOn`) — у стартовой модели прогона. Поэтому `policy` у запасной модели управляет её собственными повторами, но не тем, идти ли дальше по цепочке.

Какие категории ошибок повторяются по умолчанию, а какие нет — в [10-error-handling.md](10-error-handling.md).

### Разное поведение для разных ошибок

`perCategory` задаёт свои `retries`, `delay`, `backoff` и `maxDelay` для конкретных категорий. Типичный набор:

```php
'defaultPolicy' => [
    'retries'     => 2,
    'delay'       => 5,
    'backoff'     => 2,
    'then'        => 'fallback',
    'perCategory' => [
        // ограничение частоты: ждать дольше, но настойчивее — оно само пройдёт
        ErrorCategory::RATE_LIMIT => ['retries' => 3, 'delay' => 15],
        // таймаут медленной модели: повторять почти бессмысленно, быстрее сменить модель
        ErrorCategory::TIMEOUT    => ['retries' => 0],
        // сервер провайдера прилёг: одна быстрая повторная попытка без долгого ожидания
        ErrorCategory::SERVER_ERROR => ['retries' => 1, 'delay' => 1],
    ],
],
```

Два соседних рычага работают на уровне категорий целиком:

- **`retryOn`** — белый список: повторяем только перечисленные категории, остальные уходят на фолбэк сразу. Если список пуст, решает сама категория.
- **`stopOn`** — категории, при которых не переключаемся на другую модель, даже если `then` это разрешает. Например, `stopOn => [ErrorCategory::CONTENT_FILTER]`, если блокировка модерацией должна доходить до пользователя, а не маскироваться ответом другой модели.

Отдельной цепочки под категорию нет: цепочка одна, а различается только то, сколько раз и с какими паузами пробовать до перехода к ней.

### Сколько ждать ответа

Таймаут одного запроса задаётся у провайдера и при необходимости перекрывается моделью:

```php
'providers' => [
    'openai' => ['class' => OpenAiProvider::class, 'token' => 'sk-...', 'timeout' => 120],
],
'models' => [
    'gpt-5' => ['provider' => 'openai', 'name' => 'gpt-5', 'timeout' => 600],   // рассуждает долго
],
```

Значение уходит в транспорт как таймаут всего запроса; таймаут установки соединения берётся как `min(30, timeout)`.

Здесь легко получить неприятную арифметику: таймаут 600 секунд при `retries = 2` — это три попытки почти по десять минут, и только потом переключение на следующую модель. Поэтому кроме таймаута одного запроса есть два потолка времени: на модель и на весь вызов.

```php
'defaultPolicy' => [
    'maxWaitSeconds' => 300,      // на одну модель: её запросы и паузы между повторами
],
'maxTotalWaitSeconds' => 900,     // на весь вызов, со всеми переключениями
```

**`maxWaitSeconds` — про модель**, поэтому и живёт в политике: рядом с `retries`, `delay` и `backoff`, а значит задаётся у модели, у её провайдера или в `defaultPolicy`. Отсчёт идёт с первой попытки этой модели и **начинается заново после переключения**: если медленная стартовая модель съела свои пять минут, запасная получает свои пять, а не остаток. Когда бюджет исчерпан, повторы прекращаются и работа уходит следующей модели цепочки.

**`maxTotalWaitSeconds` — про вызов целиком**, поэтому это ключ каталога рядом с `fallback` и `maxSwitches`: он не зависит от того, какая модель работает сейчас. Отсчёт от начала вызова; когда бюджет исчерпан, прекращаются и повторы, и переключения — возвращается последняя ошибка.

Оба потолка учитывают всё время вызова, а не только паузы, и ни один не прерывает уже идущий запрос — это работа `timeout`. Поэтому фактическая длительность может превысить потолок на время последнего запроса.

Как это выглядит при `timeout = 180` и `retries = 2`:

```
glm     попытка 1 → timeout (180 с)
        пауза 5 с
glm     попытка 2 → timeout (180 с)      ← модель заняла 365 с, бюджет 300 исчерпан
                                          повтор отменён, хотя retries это разрешал
mimo    попытка 1 → timeout (180 с)      ← переключение: у mimo свои 300 с
        пауза 5 с
mimo    попытка 2 → успех ✓              ← всего ~730 с, общий потолок 900 не задет
```

## Цепочка запасных моделей

Цепочка одна на каталог и плоская: у моделей нет собственных списков продолжения. Поэтому не возникает вопроса, чей список главнее, а зацикливание невозможно — уже опробованные модели пропускаются.

```php
'fallback'    => ['glm', 'mimo', 'gpt-5-mini'],
'maxSwitches' => 2,
```

Как это выглядит, если вызывали `gpt-5` (её самой в цепочке нет — это нормально):

```
gpt-5   попытка 1 → timeout
gpt-5   попытка 2 (пауза 5с)  → empty_response
gpt-5   попытка 3 (пауза 10с) → timeout        ← три попытки: это retries = 2
glm     попытка 1 → server_error               ← переключение 1
glm     попытка 2 (пауза 5с)  → успех ✓        ← дальше прогон идёт на glm
```

Решение «переключаться или остановиться» принимает политика **стартовой** модели — той, которую выбрал вызывающий. У промежуточных звеньев цепочки `then` не спрашивают: иначе вернулся бы вопрос, чьё решение главнее.

`fallback` и `maxSwitches` отвечают на разные вопросы: первый задаёт порядок, второй — сколько шагов по этому порядку разрешено за один вызов (по умолчанию `2`). Ограничитель нужен, чтобы упавший запрос не перебирал длинную цепочку по несколько попыток на каждую модель. Из-за него часть цепочки может остаться незадействованной: при `maxSwitches = 2` до третьей модели дело дойдёт только тогда, когда стартовая сама стоит в цепочке — иначе два шага израсходуются на первые две записи. Хотите, чтобы цепочка всегда проходилась целиком, — поставьте `maxSwitches` равным её длине.

Цепочку можно переопределить на конкретный вызов: `$orchestra->withFallback(['glm'], 1)`.

## Карта capture: данные, которых нет в контракте

Провайдеры кладут рядом с ответом всё больше всего: размышления рассуждающих моделей, ссылки веб-поиска, отказы модерации. Библиотека ничего не выбрасывает — сырой ответ доступен через `$response->raw()`. Но разбирать чужую структуру в прикладном коде неудобно, поэтому есть карта: она достаёт нужные поля и кладёт их под вашими именами.

```php
'capture' => [
    'reasoning'      => ['choices.0.message.reasoning_content', 'choices.0.message.reasoning'],
    'thinkingTokens' => 'usage.thinking_tokens',
],
```

```php
$response->extra('reasoning');   // то, что нашлось по первому непустому пути
```

Значение — путь или список путей; побеждает первый непустой. Список нужен, когда одно и то же поле разные шлюзы называют по-разному: приложение читает `extra('reasoning')` и при переключении между провайдерами ничего не замечает.

Для OpenAI-совместимых провайдеров встроенная карта уже покрывает `reasoning`, `annotations`, `refusal`, `citations`, `systemFingerprint` и имя апстрима. Конфиг её дополняет и перекрывает.

## Проверка конфига

Каталог проверяется целиком в момент сборки: провайдер, на который ссылается модель, должен существовать; ключи в цепочке — тоже; класс провайдера должен существовать и реализовывать `ProviderInterface`. Любое нарушение — `LlmConfigException` с понятным текстом, ещё до первого запроса.

## Что каталог умеет отдавать

```php
$registry->has('glm');                     // есть ли такая модель
$registry->normalize($fromForm, 'glm');    // привести значение к ключу, иначе взять запасной
$registry->model('glm');                   // ModelDefinition; бросит LlmConfigException, если нет
$registry->findModel('glm');               // то же, но null вместо исключения
$registry->labels();                       // ['glm' => 'GLM-4.6', ...] — для списка выбора
$registry->all();                          // все модели
$registry->byTag('fast');                  // модели с меткой
$registry->byProvider('requesty');         // модели одного транспорта
$registry->provider('requesty')->token;    // настройки транспорта
$registry->costOf('glm', 1000, 500);       // оценка стоимости по ценам каталога
```

`normalize()` особенно удобен там, где значение приходит извне — из формы или из базы: неизвестное или пустое он заменит на запасное, а известное приведёт к каноническому ключу.
