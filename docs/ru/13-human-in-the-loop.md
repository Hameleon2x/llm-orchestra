# Human-in-the-loop (пауза и возобновление)

Иногда тулза не может вернуть результат сама — ей нужно что-то извне процесса: ответ пользователя, апрув, внешнее событие. Вместо того чтобы блокировать воркер (ответ придёт *следующим* HTTP-запросом, через минуты), тулза **приостанавливает прогон**, а цикл **возобновляется**, когда ввод получен. Это паттерн elicitation / human-in-the-loop.

`Runner` остаётся stateless: отдельного «resume»-API нет. Возобновление — это просто повторный вызов `run()` с историей, в которую дописаны недостающие результаты.

## Сигнал паузы

Тулза просит паузу, вернув из `execute()` `Tool\Dto\Result::suspend()` вместо `ok()` / `error()`:

```php
public function execute(array $args): Result
{
    // валидируем аргументы и приостанавливаемся: результатом станет ответ пользователя, поданный при возобновлении
    return Result::suspend();
}
```

`Result::isSuspended()` это сообщает. Данных нет — результата ещё не существует.

## Что делает Runner

Когда вызов тулзы вернул `suspend()`, `Runner`:

1. **Не** пишет `tool`-сообщение для этого вызова — его результат придёт извне.
2. Продолжает исполнять **остальные** вызовы того же хода как обычно (не-suspend тулзы отрабатывают и получают свои `tool`-сообщения).
3. После хода, если хоть один вызов приостановлен, **останавливает цикл** и возвращает приостановленный `Agent\Dto\Result` вместо нового обращения к модели.

```php
$result = $runner->run($history, $toolbox, $systemPromptFn, $config, $emit);

if ($result->suspended) {
    // $result->pendingToolCallIds — id вызовов, чьи результаты нужно предоставить для возобновления.
    // События TOOL_CALL уже прошли, так что UI может отрисовать каждый вопрос/виджет.
    persistPending($dialogId, $result->pendingToolCallIds);
    return; // воркер завершается; прогон на паузе
}
```

`Agent\Dto\Result` несёт:

| Свойство              | Тип        | Значение                                                                                |
|-----------------------|------------|------------------------------------------------------------------------------------------|
| `$suspended`          | `bool`     | `true`, когда прогон приостановлен в ожидании внешнего ввода. `$content` / `$error` — `null`. |
| `$pendingToolCallIds` | `string[]` | id вызовов, чьи результаты нужно предоставить перед возобновлением.                       |

## Как ответ попадает назад: возобновление = повторный run

Хелпера «подставить ответ» нет — примитив это `Message::tool($toolCallId, $content)`, то же `tool`-сообщение, которое `Runner` пишет для обычных тулз. Чтобы возобновить:

1. Для **каждого** ожидающего id допиши в сохранённую историю `Message::tool($pendingId, $answer)`. `$answer` — то, что дал внешний ввод, сериализованное так, как описывает `firstUseHint()` твоей тулзы (например, `{"answer":"..."}`).
2. Снова вызови `run()` с этой историей. Модель видит ответы как результаты тулз и продолжает — ровно так, как если бы тулзы вернули их синхронно.

```php
// пользователь ответил на один из ожидающих вызовов
recordToolMessage($dialogId, $toolCallId, json_encode(['answer' => $answer], JSON_UNESCAPED_UNICODE));

// возобновляем только когда отвечены ВСЕ ожидающие вызовы (см. «правило протокола» ниже)
if (!allPendingAnswered($dialogId)) {
    return; // ждём остальные
}

$history = loadHistory($dialogId); // Message[], завершается ассистентскими tool_calls + по tool-сообщению на каждый вызов
$result  = $runner->run($history, $toolbox, $systemPromptFn, $config, $emit);
// модель продолжает с ответов
```

`loadHistory()` — твой код: обычно ты пересобираешь `Message[]` из того, как сохранил диалог ([07-history-serialization.md](07-history-serialization.md) описывает формат). `Runner` требует лишь валидную историю (следующий раздел).

## Возобновление: закрыть ожидающие вызовы перед обращением к модели

В OpenAI-совместимом формате **каждый** `tool_call` ассистентского сообщения обязан иметь парное `tool`-сообщение перед следующим `assistant`. `pendingToolCallIds` — список, потому что модель может спросить несколько вещей за ход (параллельные вызовы): храни множество, заполняй по мере ответов.

Идеально гейтить это самому не обязательно: `run()` **резюмируем**. Перед обращением к модели он дорешивает любой tool_call, у которого ещё нет `tool`-сообщения — тем же путём, что и обычный ход (обычные тулзы выполняются и дописывают результат; suspend — снова пауза). Неотвеченные вызовы бывают только у текущего незавершённого хода: завершённые ходы всегда закрыты (обычные — перед следующим assistant; обрезанный лимитом — tool-ошибками). Итого:

- **ответы подставлены → продолжаем:** в целой истории дорешивать нечего, модель продолжает с ответов;
- **возобновили рано → снова пауза:** запустишь приостановленный прогон до того, как пришли все ответы — ещё открытые suspend-вызовы просто снова приостановятся (идемпотентно), вернётся `suspended` с оставшимися id; ни ошибки, ни «битого» запроса;
- **оборвалось посреди исполнения → восстановление:** прогон, чей воркер умер после ассистентского хода, но до записи всех результатов, дорешает незавершённые вызовы и продолжит.

Оговорка для случая обрыва/краха: перевыполнение — **at-least-once**, поэтому тулзы с побочками делай идемпотентными или защищай. `TOOL_CALL` эмитится один раз, когда модель впервые запросила вызов, поэтому перевыполнение шлёт только недостающий `TOOL_RESULT` — без дублей событий вызова. (На happy-path suspend не перезапускается вообще ничего: как только ответ лёг `tool`-сообщением в историю, вызов считается отвеченным.)

## Сопоставление по id, не по порядку

`tool`-сообщения при возобновлении матчатся с вызовами по `tool_call_id`, а не по порядку. Не-suspend тулзы хода уже записали свои `tool`-сообщения; приостановленные ты дописываешь при возобновлении. Итоговый порядок (`A, B, q1, q2`, хотя ход был `A, q1, B, q2`) неважен — важно, что закрыт каждый id.

## Caveat: не ставь в один ход действие, зависящее от ожидаемого ответа

Не-suspend тулзы хода исполняются **сразу**, до того как появился ответ. Значит, модель не должна в одном ходе звать тулзу, эффект которой зависит от ещё не полученного ответа — например, `ask_user("удалять?")` вместе с `delete_record()` удалит до ответа пользователя. Такие зависящие действия — на **следующий** ход, уже с ответом. Зафиксируй это в `getDescription()` suspend-тулзы («зови отдельно»); семантическую зависимость `Runner` знать не может.

## События

Приостановленный вызов всё равно эмитит `TOOL_CALL` (чтобы UI отрисовал вопрос или виджет), но **не** `TOOL_RESULT` — результата ещё нет. При возобновлении `tool`-сообщение пишешь ты сам; синтетический `TOOL_RESULT` для него `Runner` не эмитит. См. [06-events.md](06-events.md).

## Набросок: тулза `ask_user`

```php
<?php
use Hameleon2x\Llm\Tool\AbstractTool;
use Hameleon2x\Llm\Tool\Dto\Property;
use Hameleon2x\Llm\Tool\Dto\Result;

final class AskUserTool extends AbstractTool
{
    public function getName(): string { return 'ask_user'; }

    public function getDescription(): string
    {
        return 'Задать пользователю вопрос с готовыми вариантами и дождаться выбора. '
            . 'Вызывай отдельно — не вместе с тулзами, чей эффект зависит от ответа.';
    }

    public function firstUseHint(): string
    {
        return 'ask_user возвращает {"answer": "..."} — выбранный пользователем вариант (или свободный текст). '
            . 'Продолжай с этого ответа.';
    }

    public function getParameters(): array
    {
        return [
            new Property('question', 'string', 'Текст вопроса', true),
            new Property('options', 'array', 'Варианты ответа', true, ['type' => 'string']),
        ];
    }

    public function execute(array $args): Result
    {
        $options = array_values(array_filter(array_map('strval', (array)($args['options'] ?? []))));
        if (trim((string)($args['question'] ?? '')) === '' || count($options) < 2) {
            return Result::error('нужны question и минимум 2 варианта');
        }
        return Result::suspend(); // ответ будет подан при возобновлении
    }

    public function shouldDisplay(array $args): bool { return true; }
}
```

Сохранение, гейт «все ли отвечены» и UI каждого вопроса — на стороне приложения; пакет даёт сигнал паузы, список ожидающих id и stateless-возобновление.
