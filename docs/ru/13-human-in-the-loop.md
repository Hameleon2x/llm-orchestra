# Human-in-the-loop (пауза и возобновление)

Иногда инструмент не может вернуть результат сам — ему нужно что-то извне процесса: ответ пользователя, согласование, внешнее событие. Вместо того чтобы блокировать воркер (ответ придёт *следующим* HTTP-запросом, через минуты), инструмент **приостанавливает прогон**, а цикл **возобновляется**, когда ввод получен. Это паттерн elicitation / human-in-the-loop.

`Runner` остаётся stateless: отдельного API возобновления нет. Возобновление — это повторный вызов `run()` с историей, в которую дописаны недостающие результаты.

## Сигнал паузы

Инструмент просит паузу, вернув из `execute()` `Tool\Dto\Result::suspend()` вместо `ok()` или `error()`:

```php
public function execute(array $args): Result
{
    // проверяем аргументы и приостанавливаемся: результатом станет ответ пользователя, поданный при возобновлении
    return Result::suspend();
}
```

`Result::isSuspended()` это сообщает. Данных нет — результата ещё не существует.

## Что делает Runner

Когда вызов инструмента вернул `suspend()`, `Runner`:

1. **Не** пишет `tool`-сообщение для этого вызова — его результат придёт извне.
2. Продолжает исполнять **остальные** вызовы того же хода как обычно (не приостановленные инструменты отрабатывают и получают свои `tool`-сообщения).
3. После хода, если хоть один вызов приостановлен, **останавливает цикл** и возвращает приостановленный `Agent\Dto\Result` вместо нового обращения к модели.

```php
$result = $runner->run($history, $toolbox, $systemPromptFn, $config, $emit);

if ($result->suspended) {
    // $result->pendingToolCallIds — id вызовов, чьи результаты нужно предоставить для возобновления.
    // События TOOL_CALL уже прошли, так что интерфейс может отрисовать каждый вопрос или виджет.
    persistPending($dialogId, $result->pendingToolCallIds);
    return; // воркер завершается; прогон на паузе
}
```

В `Agent\Dto\Result` это видно по двум полям: `$suspended` равен `true` (а `$content` и `$error` — `null`), и `$pendingToolCallIds` содержит id вызовов, результаты которых нужно предоставить перед возобновлением. Причина остановки дублируется в `$finish` — там будет `Finish::SUSPENDED`.

## Как ответ попадает назад: возобновление — это повторный run

Отдельного хелпера «подставить ответ» нет — примитив это `Message::tool($toolCallId, $content)`, то же `tool`-сообщение, которое `Runner` пишет для обычных инструментов. Чтобы возобновить:

1. Для **каждого** ожидающего id допишите в сохранённую историю `Message::tool($pendingId, $answer)`. `$answer` — то, что дал внешний ввод, сериализованное так, как описывает `firstUseHint()` вашего инструмента (например, `{"answer":"..."}`).
2. Снова вызовите `run()` с этой историей. Модель видит ответы как результаты инструментов и продолжает — ровно так, как если бы инструменты вернули их сразу.

```php
// пользователь ответил на один из ожидающих вызовов
recordToolMessage($dialogId, $toolCallId, json_encode(['answer' => $answer], JSON_UNESCAPED_UNICODE));

// возобновляем, только когда отвечены ВСЕ ожидающие вызовы (см. «правило протокола» ниже)
if (!allPendingAnswered($dialogId)) {
    return; // ждём остальные
}

$history = loadHistory($dialogId); // Message[], завершается ассистентскими tool_calls + по tool-сообщению на каждый вызов
$result  = $runner->run($history, $toolbox, $systemPromptFn, $config, $emit);
// модель продолжает с ответов
```

`loadHistory()` — ваш код: обычно история пересобирается в `Message[]` из того, как сохранён диалог ([07-history-serialization.md](07-history-serialization.md) описывает формат). `Runner` требует лишь валидную историю (следующий раздел).

## Возобновление: закрыть ожидающие вызовы перед обращением к модели

В OpenAI-совместимом формате **каждый** `tool_call` ассистентского сообщения обязан иметь парное `tool`-сообщение перед следующим `assistant`. `pendingToolCallIds` — список, потому что модель может спросить несколько вещей за ход (параллельные вызовы): храните множество и заполняйте его по мере ответов.

Гейтить это самостоятельно не обязательно: `run()` **резюмируем**. Перед обращением к модели он дорешивает любой `tool_call`, у которого ещё нет `tool`-сообщения, — тем же путём, что и обычный ход (обычные инструменты выполняются и дописывают результат; приостановленные — снова приостанавливаются). Неотвеченные вызовы бывают только у текущего незавершённого хода: завершённые ходы всегда закрыты (обычные — перед следующим `assistant`; обрезанный лимитом — ошибками в `tool`-сообщениях). Итого:

- **ответы подставлены → продолжаем:** в целой истории дорешивать нечего, модель продолжает с ответов;
- **возобновили рано → снова пауза:** если запустить приостановленный прогон до того, как пришли все ответы, ещё открытые приостановленные вызовы просто приостановятся снова (идемпотентно), вернётся `suspended` с оставшимися id; ни ошибки, ни битого запроса;
- **оборвалось посреди исполнения → восстановление:** прогон, чей воркер умер после ассистентского хода, но до записи всех результатов, дорешает незавершённые вызовы и продолжит.

Оговорка для случая обрыва или краха: перевыполнение — **at-least-once**, поэтому инструменты с побочными эффектами делайте идемпотентными или защищайте. `TOOL_CALL` эмитится один раз, когда модель впервые запросила вызов, поэтому перевыполнение шлёт только недостающий `TOOL_RESULT` — без дублей событий вызова. (На нормальном пути после паузы не перезапускается ничего: как только ответ лёг `tool`-сообщением в историю, вызов считается отвеченным.)

## Что начинается заново при возобновлении

Возобновление — это новый вызов `run()`, поэтому всё, что считается «за прогон», отсчитывается с нуля:

- **срок прогона** (`deadlineSeconds`), **потолок вызова** (`maxTotalWaitSeconds` каталога) и **потолок модели** (`maxWaitSeconds` политики) — так и задумано: пауза ждёт человека и может длиться часами, засчитывать её в бюджет модели незачем;
- **бюджет вызовов инструментов** (`maxToolCalls`) — диалог с тремя паузами может исполнить до трёх бюджетов подряд. Если нужен потолок на весь диалог, считайте его сами и передавайте остаток в `Config::$maxToolCalls`;
- **счётчики результата**: `turnsUsed`, `toolCallsUsed`, `usage`, `attempts` относятся к текущему отрезку, а не ко всему диалогу. Суммируйте их на своей стороне, если показываете общую стоимость.

Что переносится между отрезками — только история сообщений: она и есть состояние прогона.

## Сопоставление по id, не по порядку

`tool`-сообщения при возобновлении сопоставляются с вызовами по `tool_call_id`, а не по порядку. Не приостановленные инструменты хода уже записали свои `tool`-сообщения; приостановленные вы дописываете при возобновлении. Итоговый порядок (`A, B, q1, q2`, хотя ход был `A, q1, B, q2`) неважен — важно, что закрыт каждый id.

## Не ставьте в один ход действие, зависящее от ожидаемого ответа

Не приостановленные инструменты хода исполняются **сразу**, до того как появился ответ. Значит, модель не должна в одном ходе звать инструмент, эффект которого зависит от ещё не полученного ответа: `ask_user("удалять?")` вместе с `delete_record()` удалит запись до ответа пользователя. Такие зависящие действия — на **следующий** ход, уже с ответом. Зафиксируйте это в `getDescription()` приостанавливающего инструмента («вызывай отдельно»); семантическую зависимость `Runner` знать не может.

## События

Приостановленный вызов всё равно эмитит `TOOL_CALL` (чтобы интерфейс отрисовал вопрос или виджет), но **не** `TOOL_RESULT` — результата ещё нет. При возобновлении `tool`-сообщение пишете вы сами; синтетический `TOOL_RESULT` для него `Runner` не эмитит. См. [06-events.md](06-events.md).

## Набросок: инструмент `ask_user`

```php
<?php
use Hameleon2x\Llm\Tool\AbstractTool;
use Hameleon2x\Llm\Tool\Dto\Property;
use Hameleon2x\Llm\Tool\Dto\Result;

final class AskUserTool extends AbstractTool
{
    public function getName(): string { return 'ask_user'; }

    public function getDescription(): string
    {
        return 'Задать пользователю вопрос с готовыми вариантами и дождаться выбора. '
            . 'Вызывай отдельно — не вместе с инструментами, чей эффект зависит от ответа.';
    }

    public function firstUseHint(): string
    {
        return 'ask_user возвращает {"answer": "..."} — выбранный пользователем вариант (или свободный текст). '
            . 'Продолжай с этого ответа.';
    }

    public function getParameters(): array
    {
        return [
            new Property('question', 'string', 'Текст вопроса', true),
            new Property('options', 'array', 'Варианты ответа', true, ['type' => 'string']),
        ];
    }

    public function execute(array $args): Result
    {
        $options = array_values(array_filter(array_map('strval', (array)($args['options'] ?? []))));
        if (trim((string)($args['question'] ?? '')) === '' || count($options) < 2) {
            return Result::error('нужны question и минимум 2 варианта');
        }
        return Result::suspend(); // ответ будет подан при возобновлении
    }

    public function shouldDisplay(array $args): bool { return true; }
}
```

Сохранение, проверка «все ли отвечены» и интерфейс каждого вопроса — на стороне приложения; пакет даёт сигнал паузы, список ожидающих id и stateless-возобновление.
