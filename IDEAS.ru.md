## Кейс 4. `arguments`-объект ломает следующий запрос — [к рассмотрению], реальная, достижимая

**Суть.** Часть шлюзов возвращает `function.arguments` не строкой, а уже объектом. Парсер терпим
(инструмент отработает), но обратно в историю вызов уходит тем же массивом, а API требует в
`function.arguments` **строку** → на втором обороте OpenAI отвечает `400 Invalid type for ...
arguments: expected a string, but got an object` (категория `bad_request`: ни повтора, ни фолбэка,
прогон умирает). Со стороны выглядит как «библиотека собрала кривой запрос».

**Было → стало.** При сериализации вызова обратно на провод привести аргументы к строке:
`is_string($args) ? $args : json_encode($args, JSON_UNESCAPED_UNICODE)`.

**Где ставить.** Лучше в `ToolCallFactory::toArray()` (src/Factory/ToolCallFactory.php:12-19) — это
единственный общий шлюз сериализации для ВСЕХ провайдеров, включая сторонние. Дополнительно можно в
`OpenAiProvider::parseToolCalls()` (src/Provider/OpenAiProvider.php:217), чтобы инвариант самого
`ToolCall` («arguments — JSON-строка», ToolCall.php:15) был честным. `is_string`-гард исключает
двойное экранирование даже при обеих правках.

**Зачем.** Один кривой ответ шлюза сейчас роняет весь прогон без диагностики; фикс в одну строку
делает библиотеку устойчивой к вольностям провайдеров.

**Тест.** Нет ни одного (объектные `arguments` не шлёт ни один тест). Добавить.

## Кейс 5. Отказ модерации трактуется как `EMPTY_RESPONSE` — [к рассмотрению], реальная

**Суть.** `OpenAiProvider::parse()` при пустом `content` безусловно бросает `EMPTY_RESPONSE`, не
глядя на уже разобранный `finish_reason` (src/Provider/OpenAiProvider.php:133 разбирает, а проверка
пустоты на :146-152 его игнорирует). Ответ `{"content":"","finish_reason":"content_filter"}` уходит
как `EMPTY_RESPONSE`, а эта категория **ретраится и фолбэчится** (ErrorCategory.php:62), да ещё
считается «обрывом связи» (:84-88). Итог: детерминированный отказ модерации получает 2 бессмысленных
повтора (паузы 5 и 10 с), потом ту же судьбу на следующей модели цепочки. Категория `CONTENT_FILTER`
в библиотеке есть и спроектирована ровно наоборот (не ретраить, не переключать, ErrorCategory.php:75),
но провайдер её **никогда не выставляет** — она ставится только по текстовым маркерам в ErrorMapper.

**Было → стало.** В `parse()` между разбором `finish_reason` и проверкой пустоты добавить:
`if ($finishReason === 'content_filter' || isset($message['refusal'])) → CONTENT_FILTER`.

**Зачем.** Блокировка модерацией детерминирована — повтор и другая модель не помогут; тратятся
секунды и деньги. Категория под это уже есть, её просто нужно активировать. `isset($message['refusal'])`
корректно: OpenAI в норме шлёт `"refusal": null` (isset=false), при отказе — строку (isset=true).

**Оценка.** Точечно, в русле уже заложенной семантики. Не изменение ради изменения.

## Кейс 6, п.4. Слишком широкий текстовый маркер `does not exist` — [к рассмотрению], реальная

**Суть.** Ошибка запроса `400 «tool_choice function 'search' does not exist»` сейчас получает
категорию `MODEL_UNAVAILABLE` (фолбэкается), потому что текст матчит маркер `'does not exist'`
(src/Error/ErrorMapper.php:44), и `textOverridesStatus(400, …)` разрешает переопределить статус.
Но это ошибка запроса (tool_choice ссылается на инструмент, которого нет) — перебор моделей
бесполезен, все модели отвергнут тот же кривой запрос. Должно быть `bad_request` (без фолбэка).
Корень — маркер `'does not exist'` ловит и «tool_choice … does not exist» (ошибка запроса), и
«The model X does not exist» (настоящая недоступность), но настоящая недоступность и так приходит
статусом 404 → `MODEL_UNAVAILABLE`, т.е. текстовый маркер для неё избыточен, а для 400 — вреден.
То же у `'deprecated'` (:46).

**Было → стало.** Убрать `'does not exist'`/`'deprecated'` из текстовых маркеров (настоящий случай
покрыт статусом 404), либо ограничить их срабатывание статусом 404/`model_not_found`. Если какой-то
шлюз отдаёт «model does not exist» телом 200 без статуса — компенсировать узким маркером
`'model does not exist'`/`'no such model'`.

**Зачем.** Сейчас кривой tool_choice впустую гоняет всю цепочку моделей. Тестов на текущее поведение
`'does not exist'`→`MODEL_UNAVAILABLE` нет — риск сломать зелёное минимален.

## Кейс 6, п.2 (остаток). 402 текстом без статуса — [к рассмотрению], низкий приоритет

**Суть.** Основной случай уже закрыт (402 по статусу → `MODEL_UNAVAILABLE`, фолбэкается). Остался
краевой путь: если «Insufficient credits» приедет телом HTTP-200 без числового статуса/кода, то
категория станет `SERVER_ERROR` (ErrorMapper.php:119-120) — итог всё равно уводит на другую модель,
просто с лишними ретраями. При желании добавить узкий маркер `'insufficient credits'`/
`'insufficient_credits'` → `MODEL_UNAVAILABLE`. Реалистичный путь OpenRouter (статус 402 или числовой
`code`) уже корректен, поэтому не срочно.

## Кейс 10. Проводной формат OpenAI живёт выше слоя провайдера

**Дешёвая часть (доки) — [к рассмотрению], стоит взять.** `docs/12-custom-provider.md` обещает
провайдера «со своим форматом сообщений/инструментов», но `Message::$content` — `?string` (нет блоков
контента), `Message::$toolCalls` — сырые массивы в проводной форме OpenAI, а `Runner` читает её
напрямую (`$raw['function']['name']`, `$raw['id']` — src/Agent/Runner.php:781-782, 826-834). То есть
формат OpenAI протекает выше провайдера, вопреки architecture.md. Честно записать в `docs/12` (и
оговоркой в architecture.md): канонический контракт истории — OpenAI-совместимый (это уже прямо
сказано в MessageFactory.php:8-9), кастомный провайдер маппит свой формат в него и обратно, а `Message`
ограничен текстовым `content` — thinking и картинки в историю не кладутся, только в `Response->extra`.
Кода не трогает, снимает ложное ожидание.

**Дорогая часть — [к 0.5].** `Message::$content` как `string|array`, `$toolCalls` как `ToolCall[]`,
`Runner` через объекты. Открывает мультимодальность и thinking-блоки для не-OpenAI провайдеров
(Anthropic). Ломает публичный контракт, `MessageFactory`, `Runner` и весь фронт, гоняющий историю в
текущем формате. Делать только при появлении реального потребителя такого провайдера. Сейчас —
изменение ради изменения, зафиксировать как план к 0.5.

## Кейс 3. `Message::assistant()` молча принимает объекты `ToolCall` — [к рассмотрению], латентная

**Суть.** `Message::$toolCalls` — `?array`; внутри библиотеки туда всегда кладут результат
`ToolCallFactory::toArray()` (сырые массивы), но `Response::$toolCalls` — это `ToolCall[]` (объекты),
тип у обоих `array`, поэтому `Message::assistant($content, $response->toolCalls)` проходит по типам
молча. Следствия: (1) в payload утекает служебный блок `raw` со всеми потрохами провайдера — молча,
без исключения, оплачивается в каждом обороте; (2) `findUnansweredToolCalls()` и `isFirstUse()` делают
`if (!is_array($raw)) continue;` (Runner.php:826-828, 777-779) → молча пропускают объекты →
возобновление после suspend не найдёт незакрытый вызов.

**Почему латентная.** Не срабатывает ни на внутреннем пути (Runner всегда через `toArray()`), ни на
документированном пути персистентности (доки предписывают `MessageFactory::toArray/fromArray`).
Срабатывает только на «наивном» ручном `Message::assistant($content, $response->toolCalls)`. Реальная
дыра в жёсткости публичного API, но сейчас не «горит».

**Было → стало.** Нормализовать на границе `Message` (лучше в конструкторе, тогда покрыт и
`assistant()`, и прямой `new Message`): элемент `instanceof ToolCall` → `ToolCallFactory::toArray()`.
Одна нормализация закрывает оба следствия; массивы проходят насквозь, риск нулевой. `!is_array()→continue`
в Runner можно оставить как defense-in-depth.

**Тест.** Нет (объектный путь не покрыт). Добавить.

## Кейс 2. Resume дописывает tool-ответ в конец — битый порядок уходит провайдеру — [к рассмотрению], реальная

**Суть.** `findUnansweredToolCalls()` (src/Agent/Runner.php:803-838) берёт незакрытые вызовы
последнего ассистентского хода, а `executeToolCalls()` дописывает tool-ответ **только в хвост**
истории (Runner.php:636). Если между ходом с `tool_calls` и его ответами вклинилось не-tool
сообщение (напр. `user`), провайдеру уходит `assistant(tool_calls) → user → tool` → OpenAI
`400 «An assistant message with 'tool_calls' must be followed by tool messages»` → `bad_request`,
прогон умирает. Порядок нигде не валидируется (buildRequest — простой `array_merge`). Инвариант
«завершённый ход полностью отвечён» в докблоке упомянут (Runner.php:794-799), но **не проверяется**.

**Достижимо в man** через `ask_user`: серверный гейт `AiDialogService::appendUserMessage()` пропускает
добавление при статусе DONE/ERROR и **не смотрит `pending_tool_call_ids`** — защита только клиентская
(`llm.js:431-435`). Устаревшая вкладка / гонка / прямой POST мимо JS создаёт осиротевший `user` при
открытых вопросах → после ответа пользователя история становится `assistant(tool_calls), user, tool`
→ следующий resume падает 400.

**Было → стало.**
- Пакет (главное): на входе `run()` (до цикла) — если в хвосте после последнего `assistant(tool_calls)`
  есть хоть одно НЕ-tool сообщение, вернуть `ErrorInfo(CONFIG, retryable=false)` (через путь
  `failedOnAppCode`), а не молча собирать битый запрос. Ловит оба подслучая — и «сейчас дорешаем и
  сломаем», и уже запёкшийся в БД `assistant, user, tool`. Предложенный ревью `array_splice` в
  одиночку **недостаточен** (для запёкшегося случая findUnanswered вернёт пусто).
- Доки: `docs/13` (и ru, строка 71) — явно записать, что нельзя дописывать сообщения других ролей,
  пока у последнего хода есть незакрытые вызовы; `run()` это не «чинит», это ответственность
  вызывающего. Сейчас текст создаёт ложное впечатление полной безопасности.
- man (первичная защита): в `appendUserMessage()` отклонять при непустом `meta['pending_tool_call_ids']`
  тем же `AiDialogConflictException`, что и «диалог обрабатывается». Не даёт осиротевшему сообщению
  вообще возникнуть.

**Зачем.** В редком, но достижимом сценарии диалог «умирает» от невалидной истории без диагностики.
Fail-fast с понятной ошибкой + серверный гейт надёжнее молчаливого переупорядочивания.

## Кейс 8. Nudge-реплика «от пользователя» остаётся в истории на успехе — [к рассмотрению], минорная

**Суть.** При исчерпании `maxToolCalls` Runner добавляет в историю сообщение с ролью `user`
(`limitNudgeMessage` — «Лимит исчерпан, дай итоговый ответ»). Неудачные ветки возвращают историю БЕЗ
него (`$messagesBeforeNudge`, Runner.php:449-459, 480), а успешные (Runner.php:489-505 и 507-521) — С
ним: реплика, которой пользователь не писал, попадает в возвращаемую `$result->messages`. Прямо
противоречит комментарию автора рядом (Runner.php:441-442).

**Было → стало.** На успешной ветке возвращать `$messagesBeforeNudge` + финальный ответ (симметрично
неудачной, ~2 строки на под-ветку). Порядок для провайдера не ломается.

**Влияние на man — НЕТ (man иммунен).** man не персистит `$result->messages` (историю пересобирает из
`ProcessAiItem`, а nudge Runner не эмитит). Но в пакете баг реальный: любой потребитель, сохраняющий
`$result->messages` (документированный контракт), получит лишнюю user-реплику. Дёшево, делает контракт
консистентным между ветками.

## Кейс 7. `ToolArgsGuard`: именной слой отбивает нормальный HTML — [к рассмотрению], реальная (редкая)

**Суть.** Guard помимо общего маркера `<parameter`/`<invoke` ищет теги, названные по именам
параметров тулзы, **в любом месте любого значения** (src/Tool/ToolArgsGuard.php:73-116). Если имя
параметра совпадает с именем HTML-тега (`title, data, body, code, head, link, …`), нормальный HTML в
значении даёт ложное срабатывание: тулза не исполняется, модели уходит «разложи параметры», она их и
так разложила → тот же отказ по кругу до `maxToolCalls`. Снаружи — «модель не умеет вызвать тулзу».

**Экспозиция man реальна, но узкая/редкая.** `ProposeProblemTool` (params `title, data, …`) и
`ProposeVariantTool` (`data`): `title`/`data` — валидные HTML-теги; если модель в `description`/
`solution` литерально напишет `<title>…`/`<data>…` — вызов отобьётся и виджет не нарисуется. Домен ПДР
редко цитирует HTML-теги → частота низкая, но не теоретическая.

**Было → стало (два варианта, оба строго лучше текущего).**
- Гибрид (как в ревью): именной слой срабатывает только если (A) значение целиком обёрнуто одноимённым
  тегом `^\s*<имя\b[^>]*>.*</\1>\s*$`, ИЛИ (B) в этом же вызове есть общий маркер `<parameter`/`<invoke`.
  Снимает массовые ложняки на встроенном HTML, полностью сохраняет защиту по общему маркеру. Цена:
  узкая дыра (чистый именной дамп-конкатенация без `<invoke>`) и остаточный ложняк на легитимных
  одноэлементных значениях (`body="<body>…</body>"`).
- Чище: именной слой **только при со-присутствии общего маркера** (B без A). Ложняков от именного слоя —
  ноль. Потеря почти теоретическая: аргументы приходят ИСКЛЮЧИТЕЛЬНО нативным JSON, а когда модель
  «протекает» полным вызовом, она почти всегда пишет канонический `<invoke name=…>` → общий маркер,
  ловится.

**Важно.** Фикс — в пакете (`ToolArgsGuard::findLeak`), а не только в man: исполнение (Runner) и
отрисовка виджета (`LlmToolService::detectArgsLeak`) зовут ОДИН метод, источник один — правка
применится к обоим синхронно. Нужны тесты (именной слой не покрыт вообще): ложняк на `body`/`title`/
`data`, true-positive на обёртке и на со-присутствии общего маркера, конкатенация — как известный gap.

## Часть 3, П4. Дедлайн не проверяется между вызовами тулз внутри оборота — [к рассмотрению], реальная (низкий приоритет)

**Суть.** Дедлайн проверяется только на границе оборота (Runner.php:142), а `executeToolCalls`
(Runner.php:538-640) исполняет инструменты в цикле без единой проверки — время тулз в срок не
засчитывается. Для man это НЕ теоретический пункт: `search_manual` — **вложенный полный прогон Runner**
со своим независимым дедлайном 240с (не ограничен остатком родительского 900с). Один оборот с
несколькими `search_manual` может законно съесть 240с × N. Остальные тулзы man — быстрые БД-запросы.

**Было → стало.** Проверять дедлайн **перед каждым вызовом** в `executeToolCalls`, и при истечении —
**дозакрывать оставшиеся вызовы хода** через `answerWithError` (как уже сделано для `limitExhausted`),
вернуть `Result::error(…, Finish::DEADLINE)`. Простой `return` без дозакрытия сломает инвариант «ход
полностью отвечён». Нужно пробросить в `executeToolCalls` `deadlineSeconds` и `startedAt`. Предел: уже
запущенную тулзу прервать нельзя — перерасход возможен на один вызов.

**Приоритет.** Низкий для man (родитель 900с, вложенный сам себя режет 240с, maxToolCalls ограничивает
число). Это дыра в гарантии дедлайна библиотеки, а не корректность.

## Часть 3, П6. `maxTurns = 0` даёт «успешный» прогон без обращения к модели — [к рассмотрению], эджкейс

**Суть.** При `maxTurns=0` цикл (Runner.php:138) не выполняется, управление проваливается на заглушку

заглушка вместо ответа. `fromArray` допускает 0 (`max(0,…)`, RunOptions.php:145). Опечатка в
`defaultRun` превращает агента в затычку, отрапортовавшую успех (отличить можно только по `finish`).

**Было → стало.** `max(1, (int)$config['maxTurns'])` в RunOptions.php:145. Легитимного «ноль оборотов»
в этой библиотеке нет («один оборот — один вызов модели»). `maxToolCalls=0` — оставить как есть
(осмысленно: «ответь без инструментов», всё равно даёт живой ответ модели).

**Влияние на man.** Живого бага нет (профили 120/45/14), фикс страхует от будущей опечатки.

## Кейс 12б. `Response::$exception` удалён, но не описан в UPGRADING — [к рассмотрению], к релизу

В 0.3.0 `Response::$exception` был публичным свойством; в 0.4 исключение переехало в
`$response->error->exception` (ErrorInfo.php:36). UPGRADING описывает соседние переименования Response

## Рассмотрено и решили НЕ делать (чтобы не всплывало снова)

### Кейс 6, п.1. `insufficient_quota` (429) → AUTH — [отклонено]
Предложение неработоспособно: для статуса 429 `textOverridesStatus` **by design** запрещает
переопределение текстом (ErrorMapper.php:212, тест ErrorsTest.php:69-79), плюс маркер `'insufficient_quota'`
недостижим — подстрока `quota` матчит `RATE_LIMIT` раньше, чем проверился бы AUTH. А `RATE_LIMIT` и
так фолбэкается: после ретраев работа уходит на другую модель (у неё свой ключ/биллинг) — нужный
исход. Практического дефекта нет, только несколько лишних попыток перед фолбэком.

### Кейс 6, п.3. Новая категория под «unsupported parameter» — [отклонено]
Нужное поведение («не ретраить, но фолбэчить») уже даёт `MODEL_UNAVAILABLE`; заводить новую категорию
ради него не стоит — набор категорий намеренно закрытый (завязан на `all()`, RETRYABLE/NOT_FALLBACKABLE,
тест `assertCount(14)`, валидацию `retryOn`/`stopOn`, доки). А штатное лечение «эта модель не принимает
temperature» — объявить `unsupported: [temperature]` у модели (CHANGELOG.md:21), а не чинить маппинг.

---

### Часть 3, П1. Эскалация перебирает цепочку с начала (не «вперёд от текущей») — [не делаем код]
`nextModel()` (Orchestra.php:352-367) идёт по цепочке с индекса 0, пропуская опробованные — при старте
с середины эскалация может пойти к более ранней (не более сильной) модели. Это **осознанный
задокументированный дизайн**: цепочка — плоский «список приоритетных запасных», а не «продолжение с
текущего места» (docs/ru/02:306-329). Для man даже полезно: старт с `mimo-2.5` → next = `glm-4.6`
(воркхорс, defaultModel), `maxSwitches=3` = длине цепочки. Менять код — сломать контракт. Максимум —
оговорка-ловушка в docs/02.

### Часть 3, П2. `max_attempts` считается по категории текущей ошибки — [не делаем код]
`runModel` берёт `maxAttemptsFor($error->category)` (Orchestra.php:297), поэтому «потолок» M в счётчике
«повтор N из M» может сдвинуться между повторами, если сменилась категория. Это **не «N из M при
N>M»**, а движущийся потолок, отражающий реальную политику; в момент показа всегда `attempt ≤
max_attempts` (by construction). Наименее плохой вариант (базовый `retries+1` дал бы «5 из 3» на
rate_limit). Код корректен; максимум — фраза в docs/06 (см. «заметки-ловушки»).

### Часть 3, П3. `defaultRun.params` сильнее `models.*.params`; `null` в params ≠ отмена — [не делаем код]
(а) `ResolvedCall::build` (ResolvedCall.php:61) сливает `defaultParams → model.params → request.params`,
где `request.params` (из `defaultRun` через runOptions) — сильнейший. Формально верно (defaultRun —
уровень вызова), задокументировано (docs/ru/08:115). (б) `null` в `params` = «не задано» (isset-семантика,
GenerationParams.php:40), базовый уровень протекает; отменить параметр — только через `unsupported`.
При этом в `extraParams` `null` = «удалить» (Merge.php:21-23) — два смысла null. Оба — корректный
дизайн. Для man влияния нет (ни одна модель не задаёт `params`, null не используется, `unsupported`
применён к gpt-5/o3). Код не менять; дописать заметки-ловушки в docs/02.
